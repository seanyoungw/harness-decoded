# 查询引擎：背压、重试与响应缓存

> 「只是调个 API」的模块有 4.6 万行 —— 原因如下。

> **English:** [04-query-engine.md](../04-query-engine.md)

---

## 查询引擎实际在做什么

表面工作：拿 message 列表，发给 Anthropic API，拿回响应。朴素实现十行代码。

生产里朴素实现会在每种场景下失败，因此才有约 46K 行：

| 场景 | 朴素失败 | Query Engine 对策 |
|------|----------|-------------------|
| 限流 | 直接崩 | 指数退避 + jitter |
| 过载 | 直接崩 | 可配置重试与预算 |
| 慢网络 | 永远挂 | 连接超时 + 重试 |
| 上下文过长 | 直接崩 | 触发 Memory 压缩后重试 |
| 流中断 | 把半包当完整 | 流完整性校验 |
| 并发子 agent | 竞态、重复请求 | 请求去重 |
| Token 预算超限 | 继续花钱 | 硬预算 enforcement |
| 部分可缓存响应 | 全量重打 | 语义缓存查找 |

每一行对应一类线上事故。

---

## 流式管线

非平凡请求用流式。工具调用的参数以 JSON 片段流式到达，引擎增量拼装 —— 英文版含完整 `StreamingToolCallBuffer` 示例。

**背压**：下游（终端渲染、IDE bridge）慢于 API 时，有界队列阻塞 `put()`，避免内存耗尽。见英文版 `BackpressureBuffer` 示例。

---

## 重试架构

按错误类型区分可重试与不可重试；429/529/500/503/网络/context_length 各有策略。**Jitter** 避免多 subagent 同时撞限后同步重试造成惊群。**context_length** 特殊：先压缩再重试，而非单纯 sleep。

英文版含完整 `RetryPolicy` / `RETRY_POLICIES` / `call_with_retry` 代码。

---

## Token 统计

`TokenBudget` 跟踪会话与单次调用用量，超预算抛错，避免静默超支。企业部署常配每用户/每项目上限。

---

## 响应缓存

多 agent 常重复相同工具-观察序列。缓存键 = hash(model, system, messages 至工具结果)。默认 TTL 约 5 分钟；仅内存、不跨会话持久化，避免代码变更后仍命中脏缓存。

---

## 模型路由

按请求特征在快慢模型间切换（工具-only 步 vs 综合推理）。降低成本而不牺牲关键路径质量。

英文版含 `select_model` 示意代码。

---

## 可观测性

`QuerySpan` 记录每次调用的模型、token、延迟、`stop_reason`、重试次数、缓存命中、工具名列表等，进审计与（生产环境）OTLP。

---

## 46K 行里都有什么

核心路径约数百行；其余大致包括：流式管线、重试与熔断、token 与计价表、缓存、路由、观测导出，以及**大量测试**（泄露树为生产代码，测试占比很高）。

---

## 示例中的实现

Level 1 实现重试核心与 token 统计；流式简化。Level 2 补全流式与缓存。`QueryEngine.call()` **按错误类型**决定是否重试 —— 400 重试无意义，429 有意义。
