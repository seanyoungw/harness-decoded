# ADR-003：内存压缩触发策略

**状态**：已采纳  
**日期**：2024-01

> **English:** [003-compaction-triggers.md](../../adr/003-compaction-triggers.md)

## 背景

autoCompact 需决定何时触发。候选策略：

1. **Token 数阈值** — 估计 token 超过窗口的 X% 时压缩  
2. **消息条数阈值** — 每 N 条消息压缩一次  
3. **主动压缩** — 每轮迭代后都压缩，保持上下文极小  

## 决策

采用 **上下文窗口 token 阈值的 85%**。

## 理由

**消息条数是糟糕代理。** 工具结果体量差异巨大。十次大文件 `read_file` 可能超限；五十次短工具调用可能仍安全。条数不反映真实 token。

**每轮都压太贵。** 压缩本身也是 LLM 调用；每轮都做会成倍增加长会话 API 成本。85% 阈值在「需要时才压」。

**为何 85% 而非 95%？** 压缩 prompt 也占 token。若等到 95%，压缩请求本身可能塞不进窗口。泄露源码行为分析亦指向类似阈值（约 85%）。留 15% 余量给压缩调用。

**失败计数是关键。** 没有 `MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES` 时，卡在压缩失败循环的会话会无限烧预算。三次上限是硬停止；之后 harness 报错而非静默循环。用户可开新会话换干净上下文。

## 后果

- 长会话（>100 轮）会多次压缩  
- 每次压缩可能损失部分细节（有意设计）  
- 85% 可通过环境变量 `COMPACT_THRESHOLD` 按部署调优  
- 压缩失败会计数并暴露 —— 运维可对该指标告警
