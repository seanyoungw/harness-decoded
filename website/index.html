Core Architecture

The Harness: Five Layers

Click any layer to expand its internals. The model sits below all five — it's one component, not the system.

Tool System
40+ tools · permission-gated · sandboxed execution · typed I/O
~29K lines
The largest conceptual surface in the harness. Each tool is not just a function — it's a typed, validated, permission-gated unit with its own error taxonomy. Before any tool executes, the harness checks requiredPermissions against the session's PermissionSet. Tools that write files require FS_WRITE; shell execution requires SHELL_EXEC (denied by default). The undercover.ts tool context flag suppresses AI attribution in git commits when operating in public repos. Notable tools: bash, read_file, patch_file, web_fetch, spawn_subagent, ask_user.
Query Engine
streaming · backpressure · retry logic · token accounting · response caching
~46K lines
The largest single module in the codebase at ~46K lines — which tells you how much complexity lives in "just calling an API." The engine handles streaming responses token-by-token, building tool call structures incrementally. Retry logic distinguishes retryable errors (429, 529, 500, 503) from fatal ones (400, 401). Exponential backoff with jitter on rate limits. Context-length errors trigger compaction and a single retry. Response caching deduplicates identical tool-observation sequences across subagents. Token accounting tracks per-call and per-session costs with configurable budget limits.
Memory System
autoCompact · KAIROS daemon · autoDream consolidation · checkpointing
KAIROS
Three subsystems. autoCompact triggers at ~85% context capacity, summarizing history while preserving task spec, constraints, and progress. KAIROS is a daemon that runs in a forked subprocess while the user is idle — the fork isolation ensures it cannot corrupt the live agent context. autoDream runs inside KAIROS: reads all session transcripts, merges overlapping observations, resolves contradictions (newer wins), converts tentative notes to durable facts. The result is checkpointed and loaded at next session start. A leaked internal metric: 1,279 sessions/day were burning ~250K API calls on consecutive compaction failures before the MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3 fix.
Multi-Agent Orchestrator
task decomposition · subagent spawning · fan-out patterns · result aggregation
swarm
Three coordination patterns. Sequential: default, subagents run one at a time. Parallel fan-out: independent subtasks spawn concurrently, results gathered at a barrier. Swarm: subagents dynamically spawn further subagents as they discover complexity; the orchestrator receives async reports. Each subagent receives a scoped context — task-specific, not the full session history — reducing token costs and preventing information bleed between independent subtasks. Subagent failure policies: retry, skip, or abort propagation configurable per spawn.
IDE Bridge
terminal UI (Ink) · VSCode extension · JetBrains plugin · approval protocol
bidirectional
Three implementations, one protocol. The terminal UI uses Ink (React for terminals), explaining why Claude Code's output renders as a structured UI rather than scrolling text. VSCode and JetBrains extensions communicate over a local socket. The bridge sends: file contents, cursor position, selection, active diagnostics. It receives: diffs for preview, approval requests for destructive operations, agent status. The bridge is deliberately thin — it serializes intent, but makes no AI decisions. This separation means the harness is testable headlessly.

From the Leaked Source

What we learned

Key architectural facts that emerged from the March 2026 source map leak.

⚙️
46K lines in the Query Engine
The module that "just calls the API" is the largest in the codebase. Streaming, retry, backpressure, caching, token accounting — none of it is trivial at production scale.
🔧
29K lines in the Tool System
Each of 40+ tools ships with typed schemas, permission requirements, error taxonomy, and sandbox config. "Execute bash" is 29,000 lines when done safely.
🧠
Memory runs in a fork
The KAIROS daemon runs in a forked subprocess to prevent memory consolidation from corrupting the live agent context. Process isolation as a correctness guarantee.
🕵️
Undercover Mode
undercover.ts blocks internal codenames (Capybara, Tengu) from commit messages and suppresses AI attribution when operating in public open-source repositories.
📊
250K API calls wasted daily
A comment in the source: before MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3, 1,279 sessions/day were burning ~250,000 API calls on consecutive compaction failures. Fixed in 3 lines.
🚀
Next model: Capybara
The source references beta flags for a model family codenamed Capybara with a significantly larger context window. "Fast" and "slow" variants implied by the routing logic.

Implementation

Wrapper vs Harness

The same "run an agent" task — two fundamentally different approaches. This is why production systems fail.

wrapper.py — what most people build 
# Looks fine. Falls apart in production.

def run_agent(task: str) -> str:
    messages = [
        {"role": "user", "content": task}
    ]

    while True:
        resp = client.messages.create(
            model="claude-opus-4-5",
            messages=messages,
            # no tools defined properly
            # no retry logic
            # no token budget
        )

        if resp.stop_reason == "end_turn":
            return resp.content[0].text

        # 😬 context grows until crash
        messages.append({
            "role": "assistant",
            "content": resp.content
        })
        # no permission checks
        # no audit trail
        # no max_iterations guard
        result = eval(extract_tool_call(resp))
        messages.append(result)
harness.py — what Claude Code taught us 
# Every failure mode handled explicitly.

async def run(self, task: str):
    trace = ExecutionTrace(task)
    messages = [{"role": "user", "content": task}]

    while trace.iterations < self.max_iterations:
        trace.iterations += 1

        # retry logic inside query engine
        resp = await self.query_engine.call(
            messages, tools, system, trace
        )

        if resp.stop_reason == "end_turn":
            return extract_text(resp), trace

        for block in resp.content:
            # permission check before execution
            tool = self.registry.get(block.name)
            result = await tool.execute(
                block.input,
                self.permissions  # enforced
            )
            trace.tool_calls.append(...)  # audited

        # memory compaction if needed
        messages = await self.memory.maybe_compact(
            messages
        )
