Three memory subsystems

Production harnesses split hot-path compaction (autoCompact during the agent loop) from idle consolidation (KAIROS / autoDream in an isolated process). They answer different questions: “can the next LLM call fit?” vs “what should the next session remember?”

Subsystem When it runs What it preserves
autoCompact During the loop, when estimated fill crosses ~85% of the context window Task spec, constraints, progress, open questions — as a structured summary block
KAIROS After the session, while the user is idle; runs in a forked subprocess Merges transcripts, queues durable facts; cannot corrupt live agent memory
autoDream Inside KAIROS consolidation passes Resolves contradictions (newer wins), promotes tentative notes, writes checkpoints

Failure guard: consecutive compaction

Leaked commentary described sessions stuck in repeated autoCompact failures, burning API quota. A hard cap — MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3 in teaching code — stops infinite retry loops and surfaces an error to the user. Pair with the lifecycle token chart: without compaction, cost compounds every iteration.
