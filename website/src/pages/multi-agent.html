Multi-Agent · Deep Dive

Three patterns
for parallel work

Sequential, parallel fan-out, swarm. Each solves a different problem. Most developers reach for multi-agent too early — or too late. Here's how to decide.

Heuristic from Claude Code discussions: parallel subtasks only pay off once you expect >15 sequential LLM calls for a single agent — otherwise coordination dominates.

System map

Fan-out harness topology

Orchestrator fans out to fresh harness instances (never a shared ToolRegistry). Results meet at a barrier, then a synthesizer LLM call merges them. Swarm adds recursive spawn with a hard cap.

Failure strategies (toy names)

Pick based on whether partial success is acceptable. Teaching harnesses mirror these as enums in examples.

ABORT_ON_FIRSTStop when any subagent fails — use when writes must stay consistent.
BEST_EFFORTCollect what succeeded; surface gaps in synthesis — good for research pulls.
RETRY_FAILEDRetry transient errors with backoff; cap attempts to protect budget.
REQUIRE_ALLAll subagents must succeed before merge — strictest for release gates.
Swarm watch-outs: enforce max_agents; if subtasks share >~30% of mutable context, coordination cost may exceed parallelism gains — prefer a single agent with better tools.

Decision guide

Should you go multi-agent?

Multi-agent adds coordination overhead. It pays off only when the conditions below are met.

✓ Use multi-agent when...
  • Subtasks are independent (don't share write targets)
  • Task exceeds single context window, even with compaction
  • Estimated >15 sequential LLM calls for one agent
  • Subtasks have different permission requirements
  • You need redundancy (multiple agents cross-checking)
  • Task structure can be decomposed upfront
✗ Skip multi-agent when...
  • Each step depends on the previous step's output
  • Subtasks share >30% of their context
  • Task is exploratory (structure unknown until mid-task)
  • Single agent + good tools handles it in <10 iterations
  • Coordination overhead exceeds parallelism benefit
  • You're prototyping — add complexity later

Failure modes

What goes wrong in multi-agent systems

These are not hypothetical — each has a concrete mitigation built into the harness.

Infinite spawning
A swarm subagent discovers complexity, spawns more agents, which discover more complexity. Exponential growth until resource exhaustion.
Fix: max_agents = 20 hard cap. SwarmCapacityError falls back to sequential.
Thundering herd retry
Five parallel agents all hit rate limit simultaneously, retry at the same time, hit rate limit again. Retry loop compounds.
Fix: ±50% jitter on backoff delay. Semaphore on concurrency.
Shared state race condition
Two agents write to the same file simultaneously. Second write silently overwrites the first. Task appears complete but output is wrong.
Fix: fresh harness per subagent. No shared ToolRegistry or MemorySystem.
Budget exhaustion
One runaway subagent consumes the entire session token budget, leaving the orchestrator unable to synthesize results.
Fix: per-subagent token budget proportional to task complexity.
Silent partial failure
Three of five parallel agents succeed; two fail silently. Synthesis treats partial results as complete. Bugs introduced to codebase.
Fix: FailurePolicy enum. REQUIRE_ALL for write operations. FanOutResult.error is never ignored.
Context bleed
Subagents receive the full parent session history. One subagent's tool results pollute another subagent's reasoning.
Fix: SubTaskContext scopes what each subagent receives. Explicit allowlist, not full history.
