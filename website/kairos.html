Memory System · Deep Dive

KAIROS &
autoDream

Claude Code runs a background daemon while you're idle. It reads every session transcript, resolves contradictions, promotes tentative observations to facts, and synthesizes patterns across sessions. This is how the agent gets smarter over time without a fine-tune.

Session lifecycle — click any event to expand

T+0:00
Session starts
Memory store loaded from .harness/memory.json
The harness reads the consolidated memory store produced by the last KAIROS run. Facts, patterns, and open questions are injected as a context prefix before the user's task. The agent starts with accumulated project knowledge — it doesn't have to re-discover that src/config.py is frequently modified, or that the JWT tests are flaky.
T+0:01
Agent loop runs
Tool calls, LLM iterations, context accumulates
Normal agent operation. Each tool call result and LLM response is added to the message history. If context exceeds 85%, autoCompact triggers and summarizes the history. Raw messages are checkpointed to disk before any compaction — KAIROS will need them later.
T+0:45
autoCompact triggered
Context at 87% capacity → checkpoint + LLM summarization
The memory system estimates token count. At 87%, it checkpoints the raw messages to .harness/checkpoints/checkpoint_{session_id}_{timestamp}.jsonl then calls the LLM to produce a structured summary preserving task spec, completed work, current state, open questions, and key facts. The raw messages are replaced by the summary. The agent continues.
T+1:23
Session ends (user idle)
Final result delivered. KAIROS daemon queued.
The agent delivers its final response. The harness writes the complete session transcript to the checkpoint directory and queues the KAIROS daemon. KAIROS is started as a forked subprocess — the main process exits immediately without waiting for KAIROS to complete. If KAIROS crashes, the main process is unaffected.
T+1:24
KAIROS daemon starts (forked)
Independent subprocess — cannot corrupt live context
KAIROS runs in a completely isolated process. It reads session checkpoints from disk, loads the existing memory store, and begins autoDream consolidation. It has its own API key usage, its own error handling, its own log file. A bug in KAIROS cannot affect the next session — the worst case is the memory store isn't updated.
T+1:25
autoDream: Pass 1 — Observation extraction
Session transcripts → atomic facts
Each session checkpoint is read and reduced to atomic observations. "Read src/auth.py. It uses JWT with HS256. The secret is in ENV['JWT_SECRET']." becomes three observations: src/auth.py exists, JWT algorithm: HS256, JWT_SECRET loaded from env. Observations are tagged with session ID and timestamp for provenance.
T+1:28
autoDream: Pass 2 — Deduplication
Merge with existing memory store, newer wins on conflicts
New observations are merged with the existing memory store. Identical observations are deduplicated. Near-identical observations (detected via semantic similarity) are merged into the more recent version. The "last seen" timestamp is updated for observations that recur, giving KAIROS a signal about which facts are stable vs. transient.
T+1:31
autoDream: Pass 3-5 — Contradiction, certainty, synthesis
Resolve conflicts, promote tentative facts, identify patterns
Pass 3: Contradictions are resolved (newer observation wins, old one archived with a note).
Pass 4: Tentative observations ("unclear if X") that were confirmed or denied in later sessions are promoted or retracted.
Pass 5: Patterns across sessions are synthesized — frequently-touched files, recurring errors, established conventions. These become high-confidence notes injected prominently in future session context.
T+1:35
Updated memory store written
.harness/memory.json overwritten with consolidated knowledge
KAIROS writes the updated memory store atomically (write to temp file, rename). The store now contains facts from this session merged with all previous sessions, with contradictions resolved and patterns synthesized. KAIROS exits cleanly. The checkpoint files are retained for debugging.
T+next
Next session loads consolidated memory
Agent starts with accumulated project knowledge
The next session loads the updated memory store. The agent now knows: which files change frequently, which tests are flaky, which APIs have undocumented behavior encountered in previous sessions. It doesn't have to re-discover any of this. The first tool calls can be targeted rather than exploratory. This is the compounding return on KAIROS investment.

autoDream algorithm

Five consolidation passes

autoDream is not trying to index the codebase. It's building meta-knowledge: what the agent has learned that's slow to derive and useful across many sessions.

01
Observation extraction
Session transcripts → atomic, tagged facts. Each observation carries session ID + timestamp for provenance tracking.
02
Deduplication
Merge with existing store. Identical observations deduplicated. Near-identical merged into the more recent version.
03
Contradiction resolution
When two observations conflict, newer wins. Old observation archived with resolution note for debugging.
04
Certainty promotion
Tentative notes confirmed by later sessions become facts. Unresolved tentatives keep their uncertainty marker.
05
Pattern synthesis
Cross-session patterns made explicit: frequently-modified files, recurring failures, established conventions, high-coupling modules. These become prominent context in future sessions — the agent doesn't re-derive what it already learned.

Memory store format

.harness/memory.json

The output of autoDream. Loaded at the start of every session.

.harness/memory.json 
{
  "project": "my-service",
  "last_updated": "2025-11-15T14:23:00Z",
  "session_count": 12,

  "facts": [
    {
      "content": "JWT secret loaded from ENV['JWT_SECRET'], HS256 algorithm",
      "certainty": "confirmed",
      "last_seen": "2025-11-15",
      "source_session": "session_2025-11-01"
    },
    {
      "content": "tests/test_auth.py exists and covers JWT validation",
      "certainty": "confirmed",
      "last_seen": "2025-11-12"
    }
  ],

  "patterns": [
    {
      "content": "src/config.py is a frequent change point — check early when debugging failures",
      "confidence": 0.85,
      "supporting_sessions": 8
    }
  ],

  "open_questions": [
    {
      "content": "RS256 migration: key loading code still missing?",
      "first_raised": "2025-11-01",
      "last_seen": "2025-11-12"
    }
  ]
}
